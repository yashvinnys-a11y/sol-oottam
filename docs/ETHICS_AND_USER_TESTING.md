# Ethics and User-Testing Procedure

This document is a project workflow, not institutional approval. Follow the current supervisor, university and module requirements before recruiting participants.

## Required order

1. **Freeze the candidate build.** Record the version number and commit hash. Do not change learning content during a participant session.
2. **Obtain Tamil-content sign-off.** Resolve every correction in `CONTENT_REVIEW_CHECKLIST.md`.
3. **Confirm approval requirements with the supervisor.** Ask specifically about testing with minors, parental/guardian consent, child assent, recruitment, incentives, voice processing and retention periods.
4. **Prepare participant materials.** Include an information sheet, consent form, child-friendly assent form where applicable, task script, questionnaire, withdrawal procedure and data-handling statement.
5. **Pilot with adults or peers first.** Use two or three people to identify navigation, timing, wording and technical problems without making claims about child learning outcomes.
6. **Run the approved study only after clearance.** Do not recruit children aged 7–12 before the required approval and consent process is complete.

## Data-minimisation rules

- Use a generated participant code; never enter a name, email, phone number or student number into the application.
- Do not record or upload raw microphone audio unless a separately approved protocol explicitly requires it.
- The prototype does not persist recognised transcript text. Verify this again in exported files before the study.
- Store exports in an access-controlled project folder, not in the public GitHub repository.
- Record the retention period and deletion date in the approved study documents.
- Export required data after each session, verify the file, then clear local evaluation data before the next participant.

## Suggested evaluation sequence

1. brief vocabulary pre-test using the same ten concepts without revealing game answers;
2. one supervised complete or time-limited Sol Oottam session;
3. vocabulary post-test using a parallel order;
4. short usability/engagement questionnaire;
5. two or three open questions about confusing instructions, preferred activity and technical problems;
6. observer notes on task completion, assistance and retries.

Do not describe a pre/post difference as proof of learning from a small formative sample. Report it as preliminary evidence with sample-size, novelty and testing limitations.

## Session close-out

- confirm that the participant may withdraw according to the approved process;
- export and securely name the data file using only the participant code;
- clear the browser’s local evaluation data;
- document technical incidents, especially microphone denial or speech-service failure;
- do not modify or delete inconvenient results after the fact.
